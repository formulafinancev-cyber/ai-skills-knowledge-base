# HANDOFF RULES

## Purpose
This file defines how task handoffs should work inside the Formula Finance AI Office project.

Its purpose is to preserve context, reduce information loss, and make transitions between roles cleaner and more reliable.

## Main Principle
A handoff is not just a transfer of responsibility.

A handoff is a structured transfer of:
- task context;
- current status;
- known constraints;
- expected output;
- ownership.

A weak handoff creates confusion.  
A strong handoff preserves momentum.

## Rule 1 — Handoffs must be explicit
A task should not silently drift from one role to another.

When a handoff happens, it should be clear:
- who handled it first;
- why it is being handed off;
- who should handle it next;
- what the receiving role is expected to do.

## Rule 2 — Handoff only when justified
A handoff should happen only when:
- the task exceeds the current role’s specialization;
- a different role has clearer ownership;
- the next stage of work belongs to another specialist;
- the current role can no longer add meaningful value without causing overlap.

Do not hand off tasks casually or too early.

## Rule 3 — Preserve context during handoff
A handoff should include the necessary context so the receiving role does not have to reconstruct everything from zero.

At minimum, a good handoff should preserve:
- task goal;
- current state;
- known facts;
- open questions;
- constraints;
- next expected deliverable.

## Rule 4 — Preserve ownership clarity
A handoff should not blur responsibility.

At any moment, it should remain clear:
- who owns the task now;
- who owned the previous stage;
- whether the previous role is still involved or not.

The project should avoid vague shared ownership when a clear owner is possible.

## Rule 5 — Avoid handoff loops
Tasks should not bounce repeatedly between roles without progress.

If repeated transfers happen, the Coordinator should step in and clarify:
- correct ownership;
- task decomposition;
- sequence of work;
- stopping conditions.

## Rule 6 — Split multi-stage work properly
If a task naturally includes multiple role stages, split it clearly.

Example:
- Scout identifies leads;
- CRM Agent structures pipeline handling;
- Proposal Agent prepares the commercial offer;
- Metrics / KPI Agent defines follow-up measurement logic.

Do not pretend one role should do everything if the task naturally crosses several domains.

## Rule 7 — Use handoff when specialization increases
A broad task may begin with Colleague or Coordinator.

If it becomes more specialized, it should be handed off to the correct expert role.

This is expected and healthy.

The project should not force a generalist role to continue a specialist task only because it started there.

## Rule 8 — The receiving role should receive a defined expectation
A handoff should not simply say that another role should look at the issue.

It should make the next expectation clear.

Examples of useful expectation logic:
- analyze and interpret;
- prepare a structured proposal draft;
- define CRM follow-up logic;
- evaluate regulatory impact;
- define KPI framework;
- prepare implementation logic.

## Rule 9 — Keep handoffs lightweight but complete
A handoff should be complete enough to preserve context, but not overloaded with irrelevant detail.

The goal is:
- enough information to continue well;
- not so much noise that the next role cannot identify the real task.

## Rule 10 — Coordinator manages complex handoff chains
If a task requires several consecutive specialist stages, the Coordinator should help define the sequence.

This helps preserve:
- order;
- timing;
- role clarity;
- implementation readiness.

## Rule 11 — Handoffs should support documentation
If a handoff produces stable knowledge, that knowledge should eventually become a project file.

This helps reduce repeated explanation and strengthens project memory.

## Rule 12 — Handoffs should improve project continuity
A good handoff makes the next step easier, faster, and clearer.

A bad handoff creates:
- repeated work;
- missing context;
- role confusion;
- implementation delays.

The system should prefer strong continuity over convenience.

## Final Rule
Every handoff in the project should improve:
- context preservation;
- role clarity;
- task continuity;
- output quality;
- practical execution.

Handoffs should move work forward, not scatter it.
