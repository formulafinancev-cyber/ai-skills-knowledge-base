# REPORTING STRUCTURE

## Purpose
This file defines the high-level reporting structure for the Formula Finance project.

Its purpose is to give the system a clear idea of how management reporting is conceptually organized, even before specific report templates and dashboards are fully defined.

## Main Principle
Reporting is not only a collection of documents.

It is a structured way of:
- presenting information;
- organizing views;
- connecting KPIs and numbers to decisions;
- supporting different roles (owner, management, operations) with the right information.

Reporting structure should be intentional, not accidental.

## Reporting Goals
Management reporting in Formula Finance is meant to:

- increase visibility into business performance;
- clarify what is happening financially and operationally;
- highlight important signals and trends;
- support better decisions;
- reduce chaotic and fragmented reports.

## Key Reporting Dimensions
A reporting structure should consider:

- **Time**: daily, weekly, monthly, quarterly views.
- **Audience**: owner, CEO, CFO, operations, other managers.
- **Content**: financial KPIs, operational KPIs, narrative insights.
- **Depth**: high-level summary vs detailed breakdowns.
- **Format**: dashboards, structured reports, brief summaries.

Different combinations serve different needs.

## Core Reporting Layers

### 1. High-Level Management View
This layer is designed primarily for owners and top management.

Characteristics:
- concise;
- focused on key KPIs;
- highlights major trends;
- includes high-level interpretation;
- aimed at quick, informed understanding.

### 2. Financial Detail View
This layer goes deeper into financial metrics.

Characteristics:
- structured financial KPIs;
- revenue / cost / margin views;
- plan vs fact;
- trend views;
- relevant breakdowns when needed.

It is more detailed than the high-level view, but still structured.

### 3. Operational / Performance View
This layer focuses on operational aspects of performance.

Characteristics:
- operational KPIs;
- process-related metrics;
- volume/throughput/efficiency perspectives where relevant;
- links between operations and financial results.

It helps show the “operational story” behind numbers.

### 4. KPI View
This layer focuses on KPI sets in a structured way.

Characteristics:
- grouped KPI sets;
- clear naming and definitions;
- focused around decision-making;
- aligned with the KPI dictionary.

It is the bridge between raw numbers and structured understanding.

### 5. Analytical / Insight View
This layer focuses on interpretation and insights.

Characteristics:
- narrative analysis;
- “what is happening”;
- “what might this mean”;
- “what to pay attention to”;
- “where to ask deeper questions”.

It is not only data; it is reasoning.

## Reporting Frequency
Reporting should be designed with realistic frequencies in mind:

- some views may be monthly;
- some may be weekly;
- some might be on-demand.

The structure should allow for relevant periodicity without overloading the business.

## Role Alignment
Different roles in the project connect to reporting structure in different ways:

- **Metrics / KPI Agent**: cares about consistency of metrics across views.
- **Analyst**: cares about interpretation and insight layers.
- **Proposal Agent**: cares about how reporting is described in offers.
- **Tech / DevOps**: cares about implementation feasibility and data flows.
- **Colleague**: cares about summarizing and explaining reports.
- **Coordinator**: cares about how reporting fits into overall workflows.

## Connection to KPI Dictionary
The reporting structure should be consistent with the KPI dictionary:

- KPIs used in reports should be defined;
- views should not introduce unreferenced KPI names;
- interpretation should match dictionary language.

## Future Reporting Artifacts
In future, this file can be linked to:

- specific report templates;
- dashboard views;
- owner / CEO / CFO / operations report formats;
- plan vs fact reporting structures;
- per-client reporting variants.

At this stage, it defines the conceptual framework.

## Implementation Note
This file describes structure, not implementation.

Actual implementation details (data sources, technical stack, exact layouts) can be added later in separate technical or template files.

## Final Principle
The reporting structure should help Formula Finance build coherent, understandable, and useful management reporting instead of a fragmented set of unrelated reports.

It exists to keep reporting aligned with:

- decision support;
- clarity;
- structured thinking;
- stable terminology.
