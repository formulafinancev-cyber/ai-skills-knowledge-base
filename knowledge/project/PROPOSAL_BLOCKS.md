# PROPOSAL BLOCKS

## Purpose
This file defines the core building blocks that can be used to construct proposals inside the Formula Finance project.

Its purpose is to make proposal work more structured, more reusable, and more consistent across different client situations.

## Main Principle
A proposal should not be improvised from zero every time.

It should be assembled from clear and reusable logic blocks that can be adapted to the specific client case.

## Proposal Philosophy
A strong proposal should feel:
- clear;
- relevant;
- structured;
- useful;
- business-oriented;
- calm and professional.

The goal is not to sound impressive through complexity.  
The goal is to help the client understand the problem, the proposed support, and the practical value.

## Core Proposal Blocks

### 1. Client Situation Block
This block describes the client’s current situation.

Its purpose is to show that Formula Finance understands:
- the business context;
- the current reporting or visibility problem;
- the management difficulty;
- the source of confusion or weak control.

This block should feel accurate and relevant, not generic.

### 2. Problem Definition Block
This block explains the key problem more explicitly.

Examples of problems this block may cover:
- weak reporting structure;
- fragmented financial visibility;
- unclear KPI logic;
- difficulty interpreting numbers;
- lack of regular management analytics;
- weak operational visibility.

The purpose is to make the business pain understandable.

### 3. Impact Block
This block explains why the problem matters.

Possible impact areas:
- weaker decisions;
- slower management response;
- loss of visibility;
- inability to see performance clearly;
- confusion in financial interpretation;
- low confidence in numbers.

This block helps the client understand the cost of staying in the current state.

### 4. Support Scope Block
This block explains what Formula Finance can help with.

Examples:
- management reporting structure;
- KPI definition logic;
- dashboard and analytics logic;
- planning and plan-vs-fact support;
- structuring financial visibility;
- interpretation support.

This block should connect clearly to the client’s actual situation.

### 5. Work Format Block
This block explains how the work may be organized.

Possible elements:
- diagnostic phase;
- structured setup phase;
- ongoing support phase;
- recurring analytical support;
- custom implementation logic.

This block helps the proposal feel operational, not abstract.

### 6. Deliverables Block
This block explains what the client can expect to receive.

Examples:
- reporting structure;
- KPI logic;
- analytical framework;
- dashboard logic;
- interpretation layer;
- structured recommendations;
- support materials.

Deliverables should be described clearly and realistically.

### 7. Value Block
This block explains the practical benefit to the client.

Examples of value:
- clearer management visibility;
- better understanding of business performance;
- stronger reporting structure;
- more confidence in interpretation;
- improved decision support;
- reduced chaos.

This block should remain practical, not exaggerated.

### 8. Case Relevance Block
This block connects the proposal to relevant case logic.

It may include:
- anonymized similar situations;
- comparable business problems;
- structured examples of before-and-after clarity;
- proof-oriented examples that reduce abstraction.

### 9. Pricing Logic Block
This block explains the commercial logic behind the offer.

It may refer to:
- diagnostic level;
- structured setup level;
- recurring support level;
- custom scope logic.

If exact prices are not fixed, this block should still explain the pricing category clearly.

### 10. Next Step Block
This block defines what should happen after the proposal.

Examples:
- clarification call;
- scope definition;
- diagnostic start;
- tailored estimate;
- approval of the next phase.

This block should reduce ambiguity and help move the process forward.

## Proposal Quality Rules
A proposal should avoid:
- empty marketing language;
- unnecessary complexity;
- vague claims;
- generic promises;
- excessive hype.

A proposal should instead emphasize:
- clarity;
- structure;
- relevance;
- practical usefulness;
- business logic.

## Proposal Adaptation Rule
Not every proposal needs every block in the same size.

Proposal Agent should adapt the block structure depending on:
- the client situation;
- the complexity of the case;
- the maturity of the lead;
- whether the offer is diagnostic, structured, or ongoing.
