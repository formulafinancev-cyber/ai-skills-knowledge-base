# KPI DICTIONARY

## Purpose
This file defines the concept of a KPI dictionary for the Formula Finance project.

Its purpose is to give the system a place where KPI names, meanings, and usage rules can be collected and clarified over time.

At this stage it acts as a structural and conceptual foundation for future KPI definitions.

## Main Principle
A KPI dictionary is not just a list of numbers.

It is a structured catalog that explains:
- what each KPI means;
- how it should be interpreted;
- where it comes from conceptually;
- how it should be used in communication and reporting.

The same KPI name should not mean different things in different places.

## Why a KPI Dictionary Matters
A KPI dictionary helps:
- avoid confusion about definitions;
- keep reports and dashboards consistent;
- improve interpretation quality;
- support onboarding of new users;
- support Proposal and CRM with clear wording;
- support Analyst and Metrics / KPI Agent with a stable reference.

Without a dictionary, KPIs become ambiguous and hard to trust.

## What a KPI Entry Should Include
Each KPI entry in the future may contain:

- **Name**: the name of the KPI as used in reports.
- **Description**: what it actually measures in clear language.
- **Category**: financial, operational, sales, efficiency, risk, etc.
- **Interpretation**: how to read it, what “good” and “bad” roughly mean.
- **Usage**: where it is typically shown (dashboard, report, role view).
- **Notes**: any cautions, special cases, or important context.

This structure can be refined later as the project evolves.

## Dictionary Scope
The KPI dictionary should eventually cover:

- core financial KPIs;
- core operational KPIs;
- core management KPIs;
- KPIs commonly used in proposals and reporting;
- KPIs frequently mentioned in conversations with clients.

It does not need to contain every possible metric at once.  
It can grow gradually.

## Role of Metrics / KPI Agent
The Metrics / KPI Agent should treat this file as a central reference for:

- defining KPIs clearly;
- grouping KPIs into sets;
- supporting reporting design;
- supporting interpretation guidance;
- keeping terminology consistent.

As new KPIs become important in the project, they should be added here.

## Role of Analyst
The Analyst should use the KPI dictionary to:

- align analytical explanations with established definitions;
- avoid redefining metrics independently in each analysis;
- help identify when a new KPI requires a dictionary entry.

## Role of Proposal Agent
Proposal Agent should:

- use KPI language that corresponds to the dictionary;
- avoid inventing new KPI names casually;
- help clients understand value using clearly defined metrics.

## Role of Content and Publisher
Content Strategist and Publisher can use the KPI dictionary to:

- generate educational content around specific KPIs;
- avoid inconsistent terminology;
- help explain metrics in a stable, reusable way.

## Initial State Rule
At this moment, this file may not yet contain a full KPI list.

That is acceptable.

The immediate goal is to give KPI logic a dedicated home in the project structure.

Specific KPI entries can be added later as the project and its reporting models evolve.

## Future Expansion
In future iterations this file can include:

- detailed KPI tables;
- per-KPI sections;
- by-view KPI groupings (owner view, management view, operations view);
- links to reporting structures where each KPI appears;
- thresholds and ranges where appropriate.

## Final Principle
The KPI dictionary should help Formula Finance maintain clarity and consistency in how metrics are named, described, and discussed.

It exists to make KPIs more understandable and more trustworthy, not more complicated.
