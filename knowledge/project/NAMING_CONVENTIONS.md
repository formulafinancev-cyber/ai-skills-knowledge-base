# NAMING CONVENTIONS

## Purpose
This file defines naming conventions for the Formula Finance AI Office project.

Its purpose is to keep project files readable, searchable, stable, and easy to maintain as the project grows.

## Main Principle
A filename should make its purpose obvious.

A person should be able to understand what a file is for without opening it.

## Core Naming Rules
All project filenames should be:

- clear;
- descriptive;
- stable;
- easy to search;
- easy to understand at a glance.

## Preferred Style
Use a consistent style for filenames.

Preferred principles:
- use meaningful words;
- avoid vague names;
- avoid random abbreviations unless they are already standard inside the project;
- keep names readable;
- keep naming patterns consistent across related files.

## Avoid These Names
Do not use weak filenames such as:
- final;
- final-final;
- new;
- new-version;
- temp;
- misc;
- notes;
- test;
- untitled;
- draft;
- draft2;
- file1;
- document;
- random.

These names do not create durable project structure.

## Good Naming Pattern
A good filename should usually contain:
- topic;
- role;
- function;
- or document purpose.

Examples of good logic:
- PROJECT_OVERVIEW
- MODEL_STRATEGY
- AGENT_MAP
- SHARED_RULES
- TASK_ROUTING_RULES
- SERVICES_OVERVIEW
- KPI_DICTIONARY
- LEGAL_MONITORING_SOURCES
- TEMPLATE_PROPOSAL

## Naming by Document Type
Use filenames that reflect the real purpose of the file.

### For foundation files
Use names such as:
- PROJECT_OVERVIEW
- ARCHITECTURE_OVERVIEW
- MODEL_STRATEGY
- AGENT_MAP
- FOLDER_STRUCTURE

### For rules
Use names such as:
- SHARED_RULES
- FILE_USAGE_RULES
- NAMING_CONVENTIONS
- TASK_ROUTING_RULES
- HANDOFF_RULES

### For business knowledge
Use names such as:
- COMPANY_PROFILE
- SERVICES_OVERVIEW
- PRICING
- POSITIONING
- CASE_LIBRARY

### For domain files
Use names such as:
- CRM_STRUCTURE
- ICP_DEFINITION
- KPI_DICTIONARY
- BRAND_VOICE
- REGULATORY_RISK_NOTES

### For templates
Use names such as:
- TEMPLATE_PROPOSAL
- TEMPLATE_SUMMARY
- TEMPLATE_MEETING_NOTE
- TEMPLATE_CLIENT_BRIEF

### For skills and references
Use names such as:
- SKILL_USAGE_GUIDE
- EXTERNAL_SKILLS_AND_REFERENCES
- CODING_BEHAVIOR_REFERENCES
- PROMPT_ENGINEERING_REFERENCES

## Stability Rule
Once a file becomes part of the project structure, do not rename it casually.

Rename files only when:
- the old name is misleading;
- the document purpose has significantly changed;
- the user explicitly approves the rename.

Stable naming makes project memory stronger.

## Scope Rule
One filename should point to one main subject.

If a file covers too many different topics, do not solve that by making the filename longer and longer.

Instead, create a separate file with a separate purpose.

## Searchability Rule
Filenames should help future search and scanning.

This means:
- use recognizable words;
- avoid decorative naming;
- prefer explicit names over clever names;
- name files for retrieval, not style.

## Final Rule
Every filename in the project should improve:
- clarity;
- retrieval;
- maintenance;
- long-term usability.

The naming system should reduce confusion, not create it.
